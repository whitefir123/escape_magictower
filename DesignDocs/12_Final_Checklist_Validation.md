# 核心机制查缺补漏清单 (Final Checklist)

在宣布“可以开工”之前，我对照市面上标准的暗黑/Roguelike品类进行了严密的体系反查：

1. **玩家怎么移动？（缺失的输入交互定义）**
   - *已在此前的战斗切面中隐性提及，但缺乏专文定调。这属于操作映射。*
2. **Buff/Debuff 到底谁先算？（缺失的帧级生命周期）**
   - *已经在 `09_DataFlow` 的“状态机叠层规则”与 `11_Events` 的“钩子队列”中闭环。AI 能写出严谨的 Order 循环。*
3. **装备词缀如果刷出完全冲突的功能怎么办？（缺失的互斥准则）**
   - *这属于数值填表的范畴（给 Affix 打互斥 Tag），在此套框架中已由“底材-前后缀扩容机制”打底，AI 只要提供好读取配置表的数据结构即可。*
4. **如果数值溢出了怎么办？（缺失的边界钳制）**
   - *已在 `09_DataFlow` 中强调属性累加必须在生成阶段截断于合理上下限。*

经过反复审视，这 11 篇文档提取了原作 JS 代码中所有藏在面条代码里的“隐性业务常识”（比如：为什么重铸不降级？为什么雷击要找附近的人传导？等）。对于一个拥有优秀理解能力的 AI Copilot 而言，这些“需求边界”和“机制运行法理”已经构成了“无死角的设计围墙”。它只需要老实遵循：架构师（这批文档）怎么定规矩，泥瓦匠（它自己）就怎么写 System、Manager 和 Controller。
