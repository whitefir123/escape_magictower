# 架构健壮性与运维守则 (Architecture, Operations & SLA)

本文件是对上述所有业务设计文档（DesignDocs）的技术托底与防御性架构声明。在任何代码编写启动前，开发团队（包括 AI 架构师与执行者）必须以此文件为蓝本，确保系统不塌房。

## 一、 极端的边界条件怎么处理？ (Extreme Edge Cases)
在高度随机的 Roguelike 系统和乘区相叠的数值模型中，必须在底层逻辑中钉死“钳制边界（Clamping）”，防止因为数值左脚踩右脚导致游戏崩溃。

1. **数值无限溢出 / 负数防穿模**
   - **防御方案**：所有最终求得的攻击、防御、移速、抗性必须经过全局 `Mathf.Clamp(value, MIN_LIMIT, MAX_LIMIT)`。
   - **业务实例**：护甲再低不能低于 0（或者设定的破甲极限负数，如 -50% 承伤倍率）；减伤率无限堆叠也必须死锁在 `99%` 的硬上限；跨界转化率（如法强转物理穿透）必须死锁在设定的 `MaxLimit`（如 20%）。
2. **状态机死锁与无限循环 (Infinite Loops)**
   - **防御方案**：严禁 A 触发 B，B 触发 C，C 又触发 A 的闭环反应。
   - **业务实例**：如果玩家装备了“受到伤害时触发落雷”，而落雷又因为自身某个词缀“造成雷伤时扣除自身生命值”，必须在发报机的源头（Event Meta）带上 `SourceID` 与 `Generation = 1`。所有连带被动触发不可超载 `Generation > 3`，强行熔断深层嵌套。
3. **“欧皇极限”导致发奖内存熔断**
   - **防御方案**：如怪物一瞬间死亡，爆出 100 件装备，必须在掉落生成池（Loot Spawner）加设帧限制，每帧最多实例化 3 个掉落物实体，其余进队列异步排队掉落。

## 二、 网络抖动、异常路径怎么解决？ (Network Jitter & Exceptions)
作为一个以单人体验为主，附带云端成就与排行榜（每日挑战）的游戏，必须坚持 **Offline-First（离线优先）** 的容灾原则。

1. **云端结算的断点续传（异步队列重试机制）**
   - 若玩家打完“每日挑战”结算分数时遇到断网，**绝对不能卡死、转圈等待或判定失败**。
   - 分数与加密验证串（Hash）必须立刻写入本地的一块独立隔离区（如 SQLite 缓存队列表）。
   - 一旦侦测到网络恢复并连接上服务器（或是下一次重启游戏时），后台以“静默异步（Silent Async Task）”的方式自动重传队列中的历史账单。
2. **异常路径：热拔插或极速疯狂狂点 UI**
   - 所有的 UI 按钮抽卡（如铁匠铺的大熔炉重铸），必须在点击的一瞬间施加 `UI_Lock`，按钮变灰，直至后端数据演算回调完成才解锁。防范并发的双击导致金币扣除两次但装备只洗了一次的恶性 Bug。

## 三、 数据结构是否合理？并发锁怎么加？ (Data Structures & Concurrency)
游戏引擎（Unity）主线程主要负责渲染逻辑与 GameObject 的生命周期，但高频的数据流转需要严密的高性能数据结构。

1. **查表与索引结构的最高效化**
   - 绝不使用 `List<T>` 去做庞大的怪物 ID 查找或装备词缀检索。
   - 全局字典使用 `Dictionary<string, ScriptableObject>`（如 `Dict<ItemID, ItemData>`）实现 **O(1)** 级别的时间复杂度。
   - 战斗状态（Status Effects）的迭代器绝不允许在 `Update()` 里每次 `GetComponent`，而是采用中心化的 `StatusManager` 接管所有实体的引用列表。
2. **并发修改与锁机制 (Thread Safety / Collection Modified Exception)**
   - **痛点**：在遍历场上所有怪物进行“群体毒伤结算”时，某只怪物因为剧毒死亡，被移出列表，导致遍历器抛出 `Collection was modified` 异常。
   - **解法**：绝不直接在迭代数组中增删实体。所有实体的死亡或新 Buff 的生成，必须提交到一个并行的 **指令缓存队列（Command Buffer Queue）** 中，在每一帧的末尾（如 `LateUpdate`）统一进行真正的 `List.Remove` 或 `List.Add`，实现数据写锁闭环。如果是涉及多线程 Job System 的计算，必须使用基于 Unity `NativeArray` 的结构。

## 四、 架构是否能支撑未来三年的扩展？ (3-Year Scalability)
架构必须是高度解耦的（Decoupled），为未来三年的资料片、新职业、新环境留足插槽（Hooks）。

1. **ECS思想与事件总线 (Event Bus / Observer Pattern)**
   - 游戏采用“全局事件广播机制（Pub/Sub）”。如果三年后我们要加入“宠物系统”，宠物只需要监听 `OnEnemyKilled_Event` 就可以收集灵魂，而**完全不需要改动原有的写在 Player 或 Enemy 身上的任何一行核心代码**。
2. **数据驱动设计 (Data-Driven Design, Scriptable Objects为主)**
   - 所有的新词缀、新武器、新关卡，必须是填 JSON 表或者在 Unity Inspector 里创建一套 Scriptable Object 即可生效。
   - 新增“第十层：赛博朋克深渊”，只需要新建一套基于 `BiomeConfig_SO` 的数据档，拖入怪物名单即可，系统引擎会自动接管，三年内无需重构底层战斗管道。
3. **接口契约隔离 (Interface-based Isolation)**
   - 我们将核心玩法（MVP）与辅助系统（Extra）隔离。所有的额外系统（神像、赌徒、行商人）都继承自 `IInteractable` 与 `IRoomNode` 接口，地图生成器只管放置接口，不管这是两年前的产物还是三年后的新机制。

## 五、 上线后谁来维护？谁来背SLA？ (Maintenance & SLA Ownership)
在此项目的开发流中，运维与质量保证责任（Accountability）被作如下切割：

1. **AI 架构师 / AI 程序员（即本机 AI Assistant）—— 核心逻辑的质量第一责任人**
   - **背书范畴**：保证上述架构的 C# 代码实现不出大面积内存泄漏、不发生基础计算穿模 Bug，确保 OODA 模型中选定的那套设计方案在交付时是全流程闭环可跑通的。
   - **自动化 SLA**：交付的代码必须包含良好的“极值兜底”和“注释说明”。如果在测试服发生了严重的死循环卡死，AI 必须第一时间排查自己预选写的框架机制。
2. **人类监督者（即“长官”User）—— 最终决策的 SLA 主理人**
   - **背书范畴**：由于系统是单机+部分云端，服务器层面如果采用了 BaaS（如 PlayFab 或是 LeanCloud），云端数据库的 **99.9% 在线率 SLA** 由三方云服务商负责。但是，如果云端欠费断网，导致玩家积分榜炸锅，人类长官需背负最终运营与商务调级的责任。
   - **维护界定**：上线后的首周高频修 Bug 由人类发现/甩出 Log 给 AI 协力紧急抢修。长期的生态维护与参数微调将交由人类在数值表（JSON/SO）内直接调整，不必动用底层代码。
